## 개요

본 문서는 **Custom-TA v2** 프로젝트 전반에 걸쳐 AI와 협업한 과정을 기록합니다.

본 공모전은 단순한 온라인 교육 플랫폼(LMS) 구축에 국한되지 않고, **교강사·수강생·교육 운영자 등 교육 현장 구성원들의 실질적인 문제를 날카롭게 해결하는 AI 솔루션**을 지향합니다. Custom-TA v2는 이 방향성을 정확히 겨냥하여 설계되었습니다.

아이디어 발굴 → 기획 → 설계 → 구현 → QA의 전 주기에서 AI를 적극적으로 활용하였으며, 단순 코드 생성 도구가 아닌 **공동 기획자(Co-Planner)** 로서 AI를 운용하였습니다.

## 1. 문제 정의 단계 — AI와 함께 Pain Point 도출

### 접근 방식

초기 아이디어 단계에서 팀은 교육 현장의 구조적 문제를 ChatGPT 및 Cursor AI와 반복적으로 대화하며 Pain Point를 체계화하였습니다.

**AI에게 던진 핵심 질문들:**

- *"대학 강의 환경에서 강사가 반복적으로 겪는 비효율을 열거하고 우선순위를 정해줘"*
- *"학생 입장에서 AI 튜터가 존재할 때 가장 원하는 기능 상위 5가지는?"*
- *"기존 LMS(Learning Management System)가 해결하지 못한 문제는 무엇인가?"*
- *"교육 운영자(교육 기관 관리자) 관점에서 강의 품질 관리의 어려움은 무엇인가?"*

### AI가 제안한 Pain Point 분류 (발췌)

AI는 교강사·수강생 이분법 구조를 제안하였고, 이것이 최종 README의 Pain Points 섹션 구조로 그대로 채택되었습니다:

| 관점 | 핵심 고통 | AI 분석 근거 |
|------|----------|-------------|
| 교강사 | 반복 질의 처리 비용 | 수업 시간 외 메시지 응답에 평균 주 4~6시간 소모 |
| 교강사 | 학생 이해도 파악 불가 | 수십 명 이상부터 개인화 피드백이 물리적으로 불가능 |
| 교강사 | 문제 출제 비용 | 퀘스트·시험 문제 직접 제작에 과도한 시간 소모 |
| 교강사 | 범용 AI와의 방향성 불일치 | 강사의 교육 철학·강의 자료를 반영하지 못하는 AI |
| 수강생 | 즉각적 질문 창구 부재 | 늦은 밤/주말 학습 중 막힘 → 모멘텀 손실 |
| 수강생 | 취약점 인식 부재 | 틀린 이유를 모른 채 진도만 진행 |
| 수강생 | 성취감 없는 공부 | 진도·성취도가 눈에 보이지 않아 동기부여 어려움 |

### 단순 LMS와의 차별점 (AI 분석 기반)

> *"범용 LMS는 콘텐츠 전달 도구에 불과합니다. 진정한 AI 교육 솔루션은 강사의 지식을 AI에 내재화하고, 학생 데이터를 분석하여 개인화된 개입(intervention)을 자동화해야 합니다. Custom-TA가 지향해야 할 방향은 '강사의 두 번째 뇌'입니다."* — Cursor AI 기획 세션 발췌

이 분석을 바탕으로 Custom-TA v2는 단순 자료 공유·퀴즈 플랫폼이 아닌, **강사 자료를 학습한 전용 RAG AI + 데이터 기반 학습 분석 대시보드**로 방향을 확정하였습니다.

## 2. 서비스 기획 단계 — 기능 목록 및 사용자 흐름 설계

### AI 협업 방식

**Cursor AI (Claude 기반)** 를 주 협업 도구로 사용하였습니다. 기능 명세서 초안을 AI에게 리뷰 요청하고, 누락된 엣지 케이스나 UX 흐름 오류를 사전에 검출하였습니다.

### 사용자 흐름 설계 (AI 공동 작성)

아래 흐름은 AI와의 대화에서 반복적으로 정제된 최종 결과물입니다:

**교강사 핵심 흐름:**
```
회원가입/로그인 (role: instructor)
    → 강의 생성 + 수강 코드 발급
    → 강의 자료 업로드 (주차별 태깅) → RAG 인덱싱
    → AI 초안으로 퀘스트 생성 → 검토 후 학생에게 발송
    → 대시보드에서 참여율·정답률·키워드 분석 확인
    → AI 제안 수락/거절 (취약 그룹 퀘스트 재배포)
```

**수강생 핵심 흐름:**
```
수강 코드로 강의 참가
    → AI 채팅으로 즉시 질의응답 (출처 명시)
    → 퀘스트 수행 → XP 적립
    → 취약점 패널에서 내 약점 확인
    → 성취도 시각화 (XP 바, 등급, 완료 퀘스트 수)
```

### AI가 추가 제안한 기능 (채택)

1. **O/X 인라인 퀴즈** — 채팅 응답 내에 간단한 퀴즈를 삽입해 학습 강화  
   → AI가 "Duolingo의 micro-learning 패턴을 참고하라"고 제안
2. **AI 제안 패널** — 취약 데이터 기반으로 AI가 능동적으로 퀘스트 배포를 제안  
   → AI가 "강사의 결정 피로를 줄이려면 AI가 먼저 제안하는 구조가 필요하다"고 분석
3. **XP 기반 게이미피케이션** — 퀘스트 완료 시 경험치(XP) 적립 및 등급 시각화  
   → AI가 "성취 가시화는 학습 지속률을 높이는 핵심 요소"임을 근거로 제안
4. **AI 가이드 프롬프트 설정** — 강사가 AI의 응답 방향·톤을 직접 커스터마이징  
   → AI가 "강사의 교육 철학을 AI에 주입하는 메커니즘이 없으면 범용 AI와 차별화되지 않는다"고 지적

## 3. 기술 스택 선정 — AI의 아키텍처 조언

### 프론트엔드 스택 결정

팀은 초기에 Next.js와 Vite+React 사이에서 고민하였고, AI에게 의사결정을 요청하였습니다.

**AI 분석 결과:**

> *"SPA 특성상 SEO가 불필요하고, 실시간 채팅·대시보드 중심의 앱이라면 Next.js의 SSR 오버헤드는 불필요합니다. Vite + React + TypeScript 조합이 빠른 개발 속도와 번들 성능 모두에서 최적입니다. 상태 관리는 서버 상태와 클라이언트 상태를 분리하는 아키텍처를 권장합니다."*

**최종 채택 스택:**

| 레이어 | 기술 | 선정 근거 |
|--------|------|----------|
| 빌드 도구 | Vite 6 | 빠른 HMR, 최신 번들링 |
| UI 프레임워크 | React 19 | 컴포넌트 재사용성 |
| 언어 | TypeScript 5 | 타입 안전성, 대규모 협업 |
| 스타일링 | Tailwind CSS v4 | 디자인 시스템 일관성 |
| 차트 | Recharts | React 생태계 최적 호환 |
| 아이콘 | Lucide React | 일관된 아이콘 디자인 |
| 배포 | Vercel | CI/CD 자동화, 빠른 CDN |

## 4. 디자인 시스템 — AI 주도 색상/UI 가이드라인 수립

### 색상 팔레트 결정

AI에게 "교육 플랫폼에 적합한 색상 팔레트를 제안해달라"고 요청하였을 때, AI는 다음 근거로 청록(Teal) 계열을 추천하였습니다:

> *"파란색 계열은 신뢰감과 집중력을 유도하며, 교육/테크 분야에서 사용자 친화적으로 인식됩니다. 순수 파란색보다 청록(#37b1b1)은 신선함과 성장의 이미지를 더합니다. 다크 배경(#071f1f → #0d4545)과 조합하면 고급스러우면서도 눈의 피로를 줄이는 다크 테마를 구현할 수 있습니다."*

**최종 채택 팔레트:**

```css
--color-primary:       #37b1b1;   /* 핵심 액션, 강조 */
--color-primary-hover: #2a9090;   /* 호버 상태 */
--color-bg-dark-from:  #071f1f;   /* 배경 그라디언트 시작 */
--color-bg-dark-to:    #0d4545;   /* 배경 그라디언트 끝 */
--color-card-light:    #f0fdfd;   /* 카드 배경 (라이트) */
```

### UI 컴포넌트 패턴

`guidelines/Guidelines.md`는 AI와의 대화를 통해 공동 작성된 문서입니다.  
AI에게 "React + Tailwind 프로젝트에서 팀 전체가 지켜야 할 코딩/디자인 컨벤션을 정리해달라"고 요청하였고, AI가 제안한 **Non-negotiables** 원칙(3-컬럼 레이아웃, 관심사 분리, 소형 파일 유지)이 그대로 채택되었습니다.

## 5. 구현 단계 — Cursor AI와의 실시간 코드 협업

### 협업 패턴

구현 단계에서는 **Cursor IDE의 AI 기능**을 사용하여 아래 패턴으로 협업하였습니다:

#### 패턴 1: 컴포넌트 설계 → AI 초안 → 팀 리뷰 → 적용

```
1. 팀이 기능 명세 작성 (어떤 상태, 어떤 UI, 어떤 API 연동)
2. Cursor AI에게 명세를 설명하고 컴포넌트 초안 요청
3. AI가 TypeScript + Tailwind 기반 초안 생성
4. 팀이 비즈니스 로직, 도메인 모델 기준으로 리뷰 및 수정
5. 최종 코드 커밋
```

#### 패턴 2: 버그 탐지 → AI 분석 → 근본 원인 파악

대표 사례 — **퀘스트 임시 저장 버그**:

```tsx
// 버그: onClick에 함수 참조 전달 시 SyntheticEvent가 첫 인수로 전달됨
<button onClick={handleSave}>임시 저장</button>
// handleSave(sendAfterSave = false) → SyntheticEvent는 truthy → 즉시 발송 실행
```

AI가 **React SyntheticEvent가 기본 인수로 전달되는 동작**을 즉시 지적하고 수정안을 제시:

```tsx
// AI 제안 수정안 (채택)
<button onClick={() => handleSave(false)}>임시 저장</button>
```

#### 패턴 3: 아키텍처 리팩토링 — AI 주도 모듈 분리

`InstructorDashboard.tsx`가 1,400줄을 초과했을 때 AI에게 분리 전략을 요청하였습니다.  
AI는 **UI / 훅 / 서비스 레이어** 3-tier 분리 원칙을 제안하였고, 이것이 `guidelines/Guidelines.md`의 "Project structure & boundaries" 항목으로 문서화되었습니다.

## 6. API 설계 단계 — AI와 함께 계약(Contract) 정의

### 백엔드 협업 문서 자동화

`docs/backend-issues.md`는 AI와 공동으로 작성한 **프론트-백엔드 인터페이스 명세서**입니다.  
팀이 발견한 6가지 이슈를 AI에게 설명하고, AI가 아래 형식으로 구조화하였습니다:

1. **증상** — 사용자가 경험하는 현상
2. **원인** — 프론트엔드/백엔드 중 어느 레이어의 문제인지
3. **프론트엔드 조치** — 이미 완료된 프론트 수정 사항
4. **백엔드 확인 사항** — 백엔드 팀에 요청할 검토/수정 목록
5. **기대 스키마** — 정확한 JSON 응답 형식 명시

이 구조는 AI가 "백엔드 팀이 컨텍스트 없이 읽어도 즉시 이해할 수 있는 수준으로 작성해야 한다"고 제안한 원칙에 따른 것입니다.

### 신규 API 필요 사항 도출

AI는 채팅 O/X 퀴즈의 정답률이 통계에 반영되지 않는 문제를 분석하면서, **`POST /courses/{courseId}/quiz/submit` API가 누락되어 있음**을 지적하고 Request/Response 스키마까지 설계해 주었습니다. 이 내용이 `docs/backend-issues.md` 이슈 #3에 문서화되어 백엔드 팀에 전달되었습니다.

## 7. QA 및 최종 점검 — AI 기반 이슈 트리아지

### AI 활용 QA 방법론

배포 전 최종 QA 단계에서 AI를 **테스트 케이스 생성기**로 활용하였습니다.

**예시 — 퀘스트 제출 플로우 테스트 케이스 (AI 생성):**

```
TC-001: 정상 제출
  - 전제: 학생이 미완료 퀘스트를 보유
  - 액션: 5문항 전부 응답 후 제출
  - 기대: 결과 모달 표시, XP 적립, "결과 보기" 버튼으로 전환

TC-002: 중복 제출 방지
  - 전제: 학생이 이미 완료한 퀘스트
  - 액션: 퀘스트 목록 재진입
  - 기대: "시작하기" 버튼 비노출, "결과 보기" 버튼만 표시

TC-003: 오프라인/백엔드 미지원 환경
  - 전제: 백엔드가 completed 필드를 미반환
  - 액션: 퀘스트 제출 후 새로고침
  - 기대: localStorage 폴백으로 완료 상태 유지
```

### 발견된 이슈 vs. AI 도움으로 해결된 이슈

| 이슈 | 발견 방법 | AI 기여 |
|------|---------|--------|
| SyntheticEvent 임시저장 버그 | 팀 테스트 | 근본 원인 즉시 분석 + 수정안 제시 |
| Optimistic Update 누락 | AI 코드 리뷰 | "API 응답 전 로컬 상태 선반영 필요" 제안 |
| 퀘스트 completed 필드 미반환 | 통합 테스트 | localStorage 폴백 전략 설계 |
| AI 초안 주차 필터 누락 | 사용자 피드백 | `week` 파라미터 추가 방안 설계 |
| 알림 읽음 상태 새로고침 손실 | 팀 테스트 | 백엔드 확인 사항 문서화 지원 |

## 8. AI 협업 효율성 평가

### 개발 기간 및 생산성

| 지표 | 수치 |
|------|------|
| 총 개발 기간 | 8일 (04/06 ~ 04/13) |
| 프론트엔드 컴포넌트 수 | 25+ 개 |
| AI 초안 활용 비율 (추정) | 약 60% |
| AI 리뷰로 사전 차단된 버그 (추정) | 8~10건 |
| 백엔드 명세 문서 AI 작성 기여 | 약 70% |
| 배포 환경 | Vercel (production) |

### AI 협업이 특히 효과적이었던 영역

1. **반복 패턴 구현** — 로딩/빈 상태/에러 상태를 모든 컴포넌트에 일관되게 적용
2. **문서 구조화** — 팀이 구두로 파악한 이슈를 명세서 형식으로 즉시 변환
3. **엣지 케이스 발굴** — "이 플로우에서 발생할 수 있는 엣지 케이스 5가지를 나열해달라" 유형의 질의로 테스트 커버리지 향상
4. **기술 의사결정** — 스택 선정, 아키텍처 패턴 선택 시 트레이드오프 분석 제공
5. **짧은 개발 기간 극복** — 8일이라는 촉박한 일정 안에서 AI 초안 생성으로 개발 속도 확보

### AI 협업의 한계 및 팀의 역할

AI는 **도메인 컨텍스트와 비즈니스 판단**을 제공하지 못합니다.  
아래 영역은 팀이 직접 주도하였습니다:

- 교육 현장의 실제 Pain Point 검증
- 기능 우선순위 결정 (사용자 가치 vs. 개발 비용 트레이드오프)
- 최종 UI/UX 품질 검수 (AI 초안의 접근성·가독성 보완)
- 백엔드 팀과의 커뮤니케이션 및 협상

## 9. 핵심 협업 원칙 (팀 내부 가이드)

프로젝트 진행 중 팀이 정립한 **AI 협업 원칙**입니다.

### DO

- ✅ AI에게 **명확한 컨텍스트**를 제공한다 (프로젝트 배경, 기술 스택, 제약 조건 포함)
- ✅ AI 결과물은 반드시 **팀이 리뷰하고 검증**한다 — 맹목적 수용 금지
- ✅ AI와의 대화에서 나온 결정은 **문서로 기록**한다 (본 문서, Guidelines.md, backend-issues.md)
- ✅ 반복 작업은 AI에게, **창의적 판단은 팀**에게 역할을 분리한다
- ✅ "왜 이렇게 설계했는가"를 AI에게 설명하게 하여 **근거를 확인**한다

### DON'T

- ❌ AI 코드를 리뷰 없이 그대로 프로덕션에 적용하지 않는다
- ❌ AI에게 최종 의사결정을 위임하지 않는다 — AI는 **분석과 초안 제공자**
- ❌ AI가 제안한 기능을 검증 없이 전부 구현하지 않는다 — 범위(scope) 관리는 팀의 책임

## 10. 결론

**Custom-TA v2**는 단순 LMS를 넘어 **강사의 자료를 학습한 전용 RAG AI + 데이터 기반 학습 분석 + 게이미피케이션**을 결합한 차세대 교육 솔루션입니다.

본 공모전의 심사기준인 **기술적 완성도·AI 활용 능력·기획력·창의성** 모두에서 AI 협업이 핵심 역할을 하였습니다:

- **기획 시간 단축**: Pain Point 분류, 기능 명세 구조화, 사용자 흐름 설계를 AI와 반복 정제
- **구현 품질 향상**: 컴포넌트 초안 생성 → 팀 리뷰 → 적용 사이클로 8일이라는 단기간에 25+ 컴포넌트 완성
- **협업 문서 고도화**: 백엔드 명세서, 가이드라인, 이슈 문서를 AI와 공동 작성하여 팀 간 커뮤니케이션 비용 절감
- **빠른 이슈 해결**: 버그 증상을 AI에게 설명하고 근본 원인을 빠르게 파악

결과적으로 AI 협업은 **더 적은 인원으로, 더 짧은 기간 안에, 더 높은 품질의 프로덕트**를 만드는 핵심 경쟁력이었습니다.
